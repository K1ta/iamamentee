# Description

Это учебный проект. Цель - изучение системного дизайна на практике. Что проектируем: онлайн-магазин, где пользователь может добавить
товар, найти товар по фильтрам, сделать заказ. Без лишних частей, вроде UI или авторизации. Упор на шардирование, идемпотентность,
кафку, саги.

# Project structure

Проект состоит из k8s и нескольких бэкенд сервисов, и организован как монорепа:
```
backend/ - сервисы на go
k8s/ - чистые конфиги для k8s
k8s-jobs/ - отдельные джобы для запуска и удаления, например, джобы с миграцией данных между шардами
loadtest/ - планируется для нагрузочных тестов
Makefile - часто используемые команды, например, релиз сервиса
```

Проект запускается в minikube.

## Структура бэкенд-сервиса

Общая структура проекта:
```
cmd/ - CLI команды, написаны на Cobra. Собирают зависимости, запускают аппы
internal/
    app/ - аппы для запуска. Принимают зависимости, которые нужно запустить, или закрыть перед завершением работы
    domain/ - доменные модели и интерфейсы, если они используются в нескольих местах
    infra/ - весь код для взаимодействия с инфрой (конфиги, база, elastic)
        config/ - парсит конфиг из env переменных. Собирает конфиг коннектов к разным базам в одну мапу
        messaging/ - если сервис пишет в кафку, то здесь лежит producer
        storage/ - репозитории, unit of work
    pkg/ - мелкие независимые пакеты, используемые в сервисе. Например, реализация snowflake или алгоритмов шардирования
    service/ - сервисный слой, содержит бизнес-логику и оркестрирует ее с вызовами инфры
    transport/ - входные запросы в сервис, http серверы или kafka консюмеры. В некоторых сервисах может содержать workers
    workers/ - фоновые процессы, например, outbox процессор. Запускаются из апп
migrations/ - миграции для бд
    main/ - миграции для основной базы (по соглашению postgres-0)
    sharded/ - миграции для шардов, выполняются для всех подов с лейблом app=postgres из неймспейса инфры сервиса
dev.env - конфиг, подставляется в configmap для деплойменты
Dockerfile - для запуска в кубере
main.go - создает контекст для приложения и запускает cmd.Run
```

### backend/product-management

- управляет продуктами. Публикует события по продуктам в кафку, топик product-management.product
- база products шардирована по userID. Также есть outbox для публикации в кафку - он дублируется на каждый шард
- чтобы избежать ошибок с запросами в разные шарды, работа с базой разделена на две части - UnitOfWork для операций
записи, и ProductView для операций чтения
- для работы в рамках одной транзакции используется Unit Of Work. Сервис запускается в двух деплойментах - один
для http-сервера, другой - для outbox процессора
- шарды могут мигрировать - если в конфиге указаны SHARDS и PREV_SHARDS, то операции чтения идут на новые шарды,
в случае отсутствия данных - на старые. Операции записи идут на новые шарды

### backend/products

- обеспечивает поиск по продуктам - слушает события из product-management.product, сохраняет записи в базу и
индексирует их через elastic
- имеет папку /mappings в корне сервиса - это маппинг для эластика
- база шардирована по product id. У нее так же есть режим миграции шардов, как и у product-management

### backend/orders

- управляет заказами. Запускает сагу заказа
- переходы заказа по статусам:
created - начальный статус, создается по запросу от юзера
created -> processing: запускается воркером, делает успешный запрос в product-management на создание резервации
created -> failed: запускается воркером, если количество попыток перевода в processing превысило лимит
processing -> completed: запускается по запросу, переводит order в статус completed
processing -> canceled: запускается по запросу, переводит order в статус canceled

# Decisions & context

- для k8s используются обычные конфиги, без helm и других штук, чтобы получить опыт работы с raw k8s
- сага реализуется на синхронных http-запросах для учебных целей. Но проектируется с учетом того, чтобы переехать на kafka события
